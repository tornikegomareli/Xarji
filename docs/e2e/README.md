# Manual E2E tests

Agent-driven smoke tests exercised in a real Chrome window via the Claude
chrome-MCP. Not in CI — run from the developer's machine, by an agent during
PR review and before any merge to `main`.

## When to run

| Scenario | What to run |
|---|---|
| Opening a PR for a focused feature/fix | Only the file(s) covering the surfaces the PR touches. e.g. a Dashboard-only change → `dashboard.md`. A range/dateRange change → every page file (range buttons live in every page header). |
| Pre-merge into `main` | Every file in this directory, top to bottom. The "we broke literally nothing" pass. |
| After a Codex review surfaces no-ship findings | Re-run the affected file(s) once the fix is pushed. |

The cadence is documented in `CLAUDE.md` §7.5 ("Manual E2E testing") so a
fresh session picks it up automatically.

## Prereqs

**Always test against demo mode.** Real data is hostage to whatever happens
to be in the developer's installed Xarji app on the day you run the tests —
multi-segment donut, signal triggers, multi-counterparty Income, FX
transactions, recurring subs, none of these can be relied on without demo
mode. Without it, a "passing" test is just a coincidence of today's data.

Setup:

1. `client` Vite dev server on `http://localhost:5173/`, started with
   `cd client && bun run dev`.
2. **Activate demo mode** by visiting `http://localhost:5173/?demo=1` once.
   The query strips itself and the choice persists in `localStorage`. To
   exit, visit `?demo=0` or use the TweaksPanel "Demo data" toggle.
3. Verify the sidebar shows large transaction counts (~1,200+) and the
   dashboard donut shows ≥5 colored category segments. If you see only
   "Other" or one segment, demo mode is **not** active — recheck step 2.
4. Chrome with the Claude extension connected. A fresh tab per session
   (don't reuse old tabs from other conversations).

The `service` does **not** need to be running for demo mode — `instant.ts`
skips the `/api/config` round trip when demo mode is active and uses the
in-memory store directly.

## Demo dataset reference

Generated by `client/src/dev/demoData.ts`, anchored to "now" at module
load. Deterministic per-day (mulberry32, fixed seed) so the same date always
produces the same numbers — but values shift day-to-day as the anchor
moves. Test assertions should prefer **shape** over exact monetary
amounts ("≥5 segments", "≥5 distinct counterparties") and fall back to
exact-name assertions for things known by name.

**Bank senders (3):** SOLO, TBC, BOG. All enabled. Every payment / failed
/ credit references one of these `bankSenderId`s.

**Categories (11):** the full `DEFAULT_CATEGORIES` set from
`client/src/lib/utils.ts`. Every default category has at least one
merchant generating real-looking transactions.

**Cards used:** `1423`, `8891`, `2230`. Every transaction's
`cardLastDigits` will be one of these.

**Currencies:** ~90% GEL, ~7% USD, ~3% EUR (random per-payment so the
exact ratio drifts). The Income page also has a fixed `$200 Stripe
Payout` per current month so the GEL filter has at least one non-GEL
credit to skip.

**Recurring monthly subs (current month, in date order):**
iCloud+ ₾8 (day 3), Spotify ₾17 (day 5), Netflix ₾32 (day 11), Claude ₾56
(day 15), Figma ₾42 (day 18), GitHub ₾11 (day 22). All on card `1423`,
SOLO sender. Subs scheduled after today's date are skipped for the
current month.

**Current-month signal bait (always present):**
- IKEA ₾4280 large-tx outlier (BOG, card `1423`).
- "Pulse Fitness" ₾95 brand-new merchant (absent from the prior 270 days).
- 2× Wolt declines on card `8891` within 4h of each other (yesterday).

**Income current month (always present):**
- Tech Co LLC ₾4800 × 2 (1st, 15th) salary, BOG sender.
- Mid-month freelance ~₾800-2200, BOG sender (counterparty rotates:
  Design Studio, Acme GmbH, Northwind, Studio Twelve, Civic Lab).
- Wolt refund ₾18, Carrefour refund ₾52 (SOLO).
- Mom P2P ₾300 (BOG).
- Stripe Payout $200 (BOG).
- Total counterparties this month: 6+ depending on freelance picked.

**Empty variant:** `?demo=empty` returns `[]` for every collection — for
testing empty-state UIs (Income's "No incoming transactions",
Categories' empty pie, etc.).

## How an agent runs these

1. Pick the file(s) for the PR's surface.
2. Confirm demo mode is active (see Prereqs step 3).
3. For each test (each `### T-…` block), follow the **Steps** literally.
4. Compare against **Expected**. Anything that doesn't match is a regression
   — file it, fix it, push, re-run that test.

## When to update this directory

Same PR that adds a UI surface adds a test for it. Same PR that removes a
surface removes the test. The discipline is the only thing that keeps these
docs honest — there's no CI pass to fail when they drift, only the next
manual run.

If you add a new page, create a new file: `docs/e2e/<page>.md`. Index it in
the table below so the pre-merge full-run is comprehensive.

If you change `client/src/dev/demoData.ts`, audit every file here for
assertions that referenced the changed merchant / amount / counterparty
and update them in the same commit.

## Test ID convention

Each test has an ID `T-<AREA>-<NN>`:

- `AREA` = uppercase short tag: `DASH`, `TX`, `CAT`, `MERCH`, `INC`, `MGT`, `AI`, `RANGE`.
- `NN` = two-digit zero-padded sequence within the file. Numbers don't get
  reused — if you delete a test, leave a gap (or repurpose the ID with a
  comment) so old PR references still make sense.

## Reporting a test result

When an agent reports a run, the format is:

```
T-DASH-01  ✓
T-DASH-02  ✗  expected donut to show ≥5 segments, saw 1 ("Other")
T-DASH-03  ⊘  skipped — demo mode not active, sidebar showed only 11 transactions
```

Pass / fail / skipped, with a one-line reason on fail.
